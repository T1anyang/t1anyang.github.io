MIT 6.S081 lab map 思路及解决方案

该课程的第十个实验，要求实现mmap和munmap

您可以在我的github仓库中找到我的完整解决方案。

mmap实现

这个实验要求我们实现mmap与munmap的系统调用。Linux中的mmap有十分复杂的特性，不过实验只要求我们在xv6上实现将文件映射到内存中这一个基础特性。

You should implement enough mmap and munmap functionality to make the mmaptest test program work. If mmaptest doesn't use a mmap feature, you don't need to implement that feature.

至于别的特性？那就算作Undefined Behavior吧（

首先我们要添加新的syscall, 此前实验已经做过很多次，不再赘述。

对于一个进程，我们要管理其由mmap映射进内存的文件。为此，我们引入virtual memory area的数据结构，记录地址、长度、权限、flag等信息。并且对每个进程赋予16个vma结构体。

在allocproc函数中也要对vma数组初始化。

然后我们来实现sys_mmap. 首先检查输入参数是否合法、指定的权限是否被允许。由于我们要采用lazy allocation的机制，因此在通过检查后在当前进程找一个空闲的vma写入信息，并对指定文件的引用计数加一即可。

在用户态通过mmap了映射了一段内存后，尝试读/写得到的内存时，由于我们的lazy allocation机制，操作系统并未对虚拟内存分配物理页，从而会产生page fault, 返回内核。我们需要对这种page fault进行处理。处理逻辑和lab lazy中的基本一致。同lab lazy类似地，uvmunmap和uvmcopy中也需要允许未被映射物理地址的虚拟地址存在。

在fork中，我们也要复制父进程的vma给子进程。

mmap的实现就基本完成了。

munmap实现

对于munmap, 我们支持把整段内存munmap、从开头开始的munmap和到结尾结束的munmap.

其实现思路也比较简单。如果flag是MAP_SHARED, 我们就把内存中的内容写回文件。之后，我们将指定的内存从页表中unmap. 如果一次munmap将映射的整段空间都释放，我们还需要将对应文件的计数减一。

此外，在exit中我们也要进行额外处理。用户态没有munmap的，内核要负责收尾。

至此，我们就完成了本次实验的全部工作。
